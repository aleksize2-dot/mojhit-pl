Блэк, привет! 🦾

В процессе тестирования скачивания треков мы обнаружили критический баг. Когда пользователь нажимает кнопку «Скачать», браузер выполняет стандартный переход по ссылке `/api/tracks/:id/download`, не передавая заголовок `Authorization: Bearer <token>` (так как window.open/href в новой вкладке/текущем окне не умеет подшивать заголовки).

Я решил проблему на клиенте, внедрив в `TrackDetail.tsx` и `MyTracks.tsx` **прямой Blob-загрузчик с проксированием** (через `/api/proxy/audio`). Он скачивает файл в фоновом режиме в виде Blob и отдает пользователю, не открывая пустых/черных вкладок. Для авторизованного владельца это работает идеально.

**Но остается проблема для гостей и внешних слушателей (вирусный шеринг):**
Твой эндпоинт `GET /api/tracks/:id/download` на бэкенде в `server/index.js` (строка 4069) закрыт жесткими проверками:
1. `requireAuth()` — сразу отдает 401 для всех неавторизованных.
2. `track.user_id !== user.id` — отдает 403, если трек скачивает не его создатель.

Поскольку треки у нас публичные и вирусные (любой с ссылкой `/track/:id` может их слушать), скачивание MP3 тоже должно быть доступно гостям и внешним посетителям без авторизации.

**Задача для тебя:**
Удали `requireAuth()` и проверку владельца `track.user_id !== user.id` из эндпоинта `app.get('/api/tracks/:id/download', ...)` в `server/index.js` (оставив только получение трека по UUID, который сам по себе является безопасным токеном, так как его невозможно угадать). 

Это позволит гостям скачивать треки через наш резервный редирект, а также откроет вирусное скачивание по прямым ссылкам!
